Lesson 6: Problem Solving

Code Challenge Tutorial

This lesson is going to step through an example code challenge from start to finish. We'll go over how to break down questions so they can be solved, and learn about effective googling. If you're a beginner I really do recommend sticking through it.

What do you get?

When it comes to code challenges, they can vary quite a bit. But usually you get the following info:

  • Description: The problem explained in words.
  • Argument info: What data will you be supplied with.
  • Return request: What is your function returning?
  • Examples: A few sample inputs and outputs to see what's expected.

Each of those items will be given to you for Marcy's challenges. Let's take a look at our sample problem below.

The Problem

Question 1: Ouija()

Write a function ouija that takes a String argument, word. The function should RETURN a random character from the given word each time. The case from the word character should match the return value. If a string has no valid return characters, return an empty string.

console.log(ouija('Spooky'))
console.log(ouija('Spooky'))
console.log(ouija('Spooky'))
console.log(ouija('Spooky'))

console.log(ouija('A'))
console.log(ouija('A'))

You should see something like this

Problem Solving

How should we tackle this problem? There are a billion different strategies, but here's my process:

Step 1: Understanding

The key to solving a problem is to make sure you understand the problem. Whenever you are given a task, always read it at least twice. Don't move on until you understand exactly what is being asked of you. Your goal here is to restate the problem in your own words. For this problem, I would say:

We're going to write a function called Ouija, and it only takes a single word as an argument. It will return a single random character directly from the word. If the string is all capitals, the return will be a capital, same with lowercase. If the word passed in is an empty string, then we just return it.

Step 2: Solve with words

It's time to try and find a solution. Not the solution, any solution. When solving a question for the first time, it's much easier to improve a working answer than it is to wait for the perfect answer.

Do not code yet. Think about how you would solve the problem as a human first. What I mean by that is, if I literally told you to do this without a computer, what would you do?

The human solution

If I was asked by someone to look at a word and say a random letter, what would I do? If I was handed an empty word, I'd say nothing. But if I got a real word, I would look at each letter of the word, then to "randomize" it, I would sort of ping-pong between them for a bit, then I would just say one.

In other words:

  • Read the word
  • If it's empty, say nothing
  • Otherwise, I'd look at the letters and randomly pick one
  • I would then say that letter

Convert to pseudo-code

OK! Since we found the answer as a human, that means there is a solution, and we just need to translate it to pseudo code! We still aren't coding yet, we're sticking with general jargon. No need for syntax; only rough ideas.

"Read the word"

Computers don't read with their eyes, but they can take in arguments for their functions. So this is the part where we pass in the argument. Ok, so step one is "pass in the word as an argument."

"If it's empty..."

How does a computer know a word is empty? Well, all strings have length right? Let's check if the string has a length of 0 and if it does, then let's "say nothing" by returning an empty string. That's what the prompt asked for, and the return lets us avoid the rest of the function. Second step is "Check if there's a length, return the empty string if not."

"Read it an pick a random letter"

Ok, here's our first real limitation. Computers can't just look at a word and "know" where all the letters are. They would have to iterate through it letter by letter. But, we only need to pick a single, random letter. What if we could pick a random index (ie a number) and then return whatever character is there?

Ah, but we can't just pick any random number, if we overshoot the last index, we'll return undefined. So, it sounds like what we need is the ability to pick a random number from 0 to the last index of the word.

Hm. I didn't teach that you that did I? That's ok, because the ability to "pick a random number between 0 and the last index" seems like something a computer could absolutely do. Let's assume it can for now, and move on.

"Say the letter"

This is easy, whatever number we picked randomly, we'll just return the value at that index of the string. Done! Ok, so let's see our "code" answer below compared to our "human" solution:

Human solution Pseudo Code
Read the word Pass in the word as an argument
if it's empty, say nothing If the string has no length, return the string
Otherwise, look at the letters and randomly pick one Otherwise, find a random number from 0 to the last index of the string
Say the letter Return the value at the random index of the string

Move on to solving

There you go! That's more than enough to get us going. Time can be a factor with code challenges, so all you're looking for is a rough plan.

Step 3: Make Checkpoints

Now that we have that plan, we need to set some markers so track our progress. When deciding on checkpoints, focus on coming up with little goals that will give you mental momentum. Small, but achievable. Each checkpoint should return or log some concrete value, even if it's not the final answer. Here are the ones I came up with.

Checkpoint Goal
Call the function, and always return the word We see a return value and spelled everything right
Return "" for empty words, false otherwise We know our conditional statement is correct
console.log a random, valid index if the word exists Figure out how random works with JavaScript
Return random letter Last check point means we're done!

Step 4: Code!

Pretty crazy that we don't touch the keyboard until step 4 right? Well that's because if you do it correctly, the coding isn't the hard part. If you come up with a logical solution, then the coding is only translating and debugging.

Checkpoint 1

To prove coding isn't the hard part, let's knock out our first checkpoint. We'll call the function with some different inputs so we can make sure we spelled everything correctly and have some info for the later steps. Each log is like a test, and it should tell us something about our function, I've noted the end goals next to the logs here. 

const ouija = (word) => {
  return word
}

console.log(ouija("abc")) // lowercase letter
console.log(ouija("ABC")) // uppercase letter
console.log(ouija("")) // ""
console.log(ouija("A")) // A

Some of our examples will be outputting the wrong thing for now, but that's ok.

checkpoint 2

Now, let's add a conditional check to only return the word if there's no length and false otherwise. The false won't stick around, we just need to see that we log "" OR false, you could use any other value. 

const ouija = (word) => {
  if (!word.length) {
    return word
  }
  return false
}

console.log(ouija("abc")) // lowercase letter
console.log(ouija("ABC")) // uppercase letter
console.log(ouija("")) // ""
console.log(ouija("A")) // A

Does that answer make sense to you? word.length will only be 0 if it's empty. That's also the only falsy number. Conditionals only fire for true, so that's why we use the bang operator to flip it. If that's confusing, go check out lesson 3 again. Specifically: truthy and falsy, inverting booleans, and not needing else cases with return.

checkpoint 3

We'll have to put a pin in this, because this is the checkpoint were we don't know what to do. That means it's time to Google. You may not need it for the application questions, but you it's still a crucial skill!

How to Google Questions

When it comes to googling questions, the name of the game is specificity. You want to ask very focused questions, so that you don't get lost in the internet. If our question in normal-speak is:

How do I make a random number in javascript between 0 and one less than the final length of the sting?

Our final goal is this format:

[Language] | how to | short question

The trickiest part is "shortening" our question. So "how do I make a random number" becomes "get random number", ditch the "in javascript" because we put that at the start. But "between 0 and one less than the length of the string" needs some work. Let's think about it.

"one less than the length of the string" is really just a number. So, what we're asking is "Between 0 and another number." Actually, we don't care about 0 either. We'll need it for our problem, but if we can find a random number between 2 numbers, then we could just swap the first for 0. We really only need "between 2 numbers."

Whew! Ok let's see what we got:

js how to get random number between 2 numbers

Look how much shorter that is! And just a little tip from a pro: "between 2 numbers" has a fancy name in programming, it's called a range:

js how to get random number from range

Finding the answer

Either way, we'd plug that into Google, and come up with a few search results. Don't always go for the first result, take a look at the first few titles. I'm going with the official MDN docs, which gives me this snippet of code: 

Math.floor(Math.random() * num)

Here's a fun challenge. Instead of me explaining, why don't you try reading the docs figure it out? Here's a different blog explaining .random() as well. Does the code make sense now?

Back to checkpoint 3

Our second to last checkpoint is to put that code in (broken into 2 lines for mobile readers), and then log the index temporarily to check our work. 

const ouija = (word) => {
  if (!word.length) {
    return word
  }
  let randomNum = Math.random() * word.length
  let index = Math.floor(randomNum)
  console.log('index:', index)
  return false
}

console.log(ouija("abc")) // lowercase letter
console.log(ouija("ABC")) // uppercase letter
console.log(ouija("")) // ""
console.log(ouija("A")) // A

Copying vs understanding

It's ok for copied code to confuse you for now. The only important thing is that you mostly understand it. Don't mindlessly copy things, read the code and alter it as necessary. Case in point, I had to replace num with word.length in order to get it to work.

You will be asked about your code, so be sure you can talk about any aspect of it. For instance, an acceptable level of understanding would be, "I'm not exactly sure how floor() works, but combined with random(), it gives me a random number every time."

Checkpoint 4

The last step is easy now, just return the value at the index! 

const ouija = (word) => {
  if (!word.length) {
    return word
  }
  let randomNum = Math.random() * word.length
  let index = Math.floor(randomNum)
  return word[index]
}

console.log(ouija("abc")) // lowercase letter
console.log(ouija("ABC")) // uppercase letter
console.log(ouija("")) // ""
console.log(ouija("A")) // A

Wrapping up

There's a quote I really like, "Slow is smooth and smooth is fast." I know it can be a pain to take all this time to plan when you just want to code, but that's a trap. By slowing down to take time to understand the problem and come up with a plan, you'll save time in the long run.

A note on debugging

People think of errors like a house fire: everything is chaos, and it gets solved all at once with an obvious fix. But real debugging is like a broken chain. The way to fix a chain is to start at a link you know is correct, and then inspect each one until you find the problems.

When you have a problem with your code, take a step back, find the parts that you know work, and then move on from there. That's why coding with checkpoints is so critical. It keeps problems from getting too big, because each piece only adds a little logic at a time.

Good Luck!
